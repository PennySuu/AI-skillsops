# SkillsOps 后端技术评审文档

## 0. 文档目标与边界

- 目标：将 `需求文档`、`产品交互文档`、`开发规范`映射为可执行的后端实施方案，支持技术评审与迭代排期。
- 适用范围：V1（认证、市场、详情安装、发布与审核、评分评价、基础运营统计）。
- 约束基线（来自规范）：
  - Java 17 + Spring Boot 3.3.x + MyBatis + Maven。
  - MySQL 8.0+、Redis 7.0+。
  - API-first、统一响应结构 `{success, code, message, data}`、版本前缀 `/v1`。
  - 安全：HTTPS、参数化 SQL、敏感数据脱敏、鉴权与权限分层。

## 1. 需求理解与技术映射

### 1.1 业务场景概述（技术视角）

平台围绕 Skill 资产全生命周期管理，核心链路如下：

1. 作者创建 Skill 草稿并提交审核。
2. 管理员审核通过后上架，拒绝则回退修改。
3. 用户在市场检索、查看详情并复制安装命令（平台不直接执行本地安装）。
4. 用户仅在已安装前提下评分/评论（一人一条，可更新，评论可为空）。
5. 作者对已上架 Skill 发布新版本，已安装用户在“我的安装”感知可更新状态。
6. 管理员维护分类、下架违规 Skill，并查看运营指标（总量、趋势、排行、活跃作者）。

### 1.2 功能模块拆分（产品 -> 后端模块）

- `auth-service`：注册、登录、登出、HttpOnly Cookie 会话签发/校验、CSRF Token 签发与校验、权限上下文。
- `skill-service`：Skill 元数据管理、生命周期状态机（草稿/审核中/上架/下架）。
- `review-service`：审核流、拒绝理由、审核记录。
- `version-service`：Skill 版本管理、版本唯一性校验、最新版本计算。
- `market-service`：市场检索、排序分页、详情聚合（评分、安装、版本）。
- `install-service`：安装记录、安装计数、可更新标记计算。
- `rating-service`：评分评论、去重约束、一致性聚合（均分、评论数）。
- `category-service`：分类标签 CRUD、启停管理。
- `ops-service`：运营统计聚合（趋势/排行/作者活跃度）。
- `notification-service`（V1.5 预留）：版本更新通知、站内消息。

### 1.3 核心用例与调用链

1. **市场检索与详情访问（高频读）**  
   `Client -> Gateway -> market-service -> Redis(命中) / MySQL(回源) -> Client`
2. **安装 Skill（读写混合 + 幂等）**  
   `Client -> Gateway -> install-service -> MySQL(tx) -> Redis失效 -> metric-event`
3. **评分更新（并发冲突敏感）**  
   `Client -> rating-service -> MySQL(UPSERT) -> 异步聚合评分 -> Redis失效`
4. **提交审核与审核决策（状态机）**  
   `author -> skill-service -> review-service -> skill_status更新 -> 市场可见性变更`
5. **发布新版本（一致性 + 可更新通知）**  
   `author -> version-service -> skill.latest_version更新 -> install可更新标记刷新(异步)`

## 2. 系统架构设计

### 2.1 整体架构（ASCII）

```text
[Web/Vue3]
    |
[API Gateway / BFF]
    |
    +--> [Auth Service] ------> [MySQL]
    |
    +--> [Skill Domain Services]
    |        |- skill-service
    |        |- version-service
    |        |- review-service
    |        |- market-service
    |        |- install-service
    |        |- rating-service
    |        |- category-service
    |        \- ops-service
    |                 |
    |                 +--> [Redis]
    |                 +--> [MQ(Kafka/RocketMQ)]
    |
    +--> [Observability: Prometheus + Tracing + Logs]
```

### 2.2 服务划分与调用关系

- V1 建议采用**模块化单体（Modular Monolith）**：统一部署一个 Spring Boot 应用，内部按领域模块分包，避免早期微服务拆分带来的运维/分布式复杂度。
- 模块之间通过 Service 接口调用，禁止跨模块直接访问 DAO。
- 外部扩展能力（通知、统计离线计算）通过消息事件解耦，保障后续微服务平滑拆分。

### 2.3 部署拓扑建议

- **V1（推荐）**：单体式部署（2-3 实例）+ MySQL 主从 + Redis 主从/哨兵。
  - 原因：团队规模与业务复杂度适中，发布与排障成本最低，满足首版交付速度。
- **V2+（增长后）**：按读写热点拆分为 `market/read`、`install/rating/write`、`ops/analytics` 微服务。
- Serverless 不推荐作为主路径：长连接、缓存与数据库连接池调优受限，且团队 Java 后端栈以 Spring 为主。

### 2.4 技术栈选型与理由

- **开发框架**：Spring Boot 3.3.x + Java 17 + MyBatis  
  - 理由：与规范一致；生态成熟；MyBatis 更易精细控制复杂检索 SQL 与索引命中。
- **数据库**：MySQL 8.0+  
  - 理由：事务能力强、JSON/窗口函数可支持统计查询、团队熟悉度高。
- **缓存**：Redis 7.0+  
  - 场景：市场列表、详情聚合、热门榜单、会话与限流计数器。
- **消息队列**：Kafka（优先）  
  - 理由：安装/评分/版本事件吞吐高、生态成熟、便于后续实时统计；若团队已有 RocketMQ 运维经验，可二选一。
- **搜索引擎**：V1 不强制引入 Elasticsearch  
  - 理由：当前检索维度（分类、关键词、排序）可由 MySQL + 索引覆盖；当数据量 > 100w Skill 或复杂全文检索再引入 ES。
- **其他中间件**：
  - Sentinel（限流/熔断/降级）；
  - SpringDoc OpenAPI（契约与调试）；
  - Micrometer + Prometheus + Grafana（监控）。

## 3. 数据库设计

### 3.1 ER 图描述（核心实体）

```text
user(1) ---- (N) skill(author_id)
skill(1) ---- (N) skill_version
skill(1) ---- (N) install_record(user_id, version_id)
skill(1) ---- (N) rating(user_id unique with skill_id)
skill(N) ---- (1) category
skill(1) ---- (N) audit_record
skill(1) ---- (1) skill_stat_agg
```

### 3.2 关键表结构（字段、约束、索引、查询场景）

#### 3.2.1 `user`

- 字段：`id(bigint PK)`、`username(varchar64 UNIQUE)`、`password_hash(varchar255)`、`role(tinyint)`、`status(tinyint)`、`created_at`、`updated_at`。
- 索引：
  - `uk_username(username)`：登录按账号查询。
  - `idx_role_status(role,status)`：后台按角色/状态过滤。

#### 3.2.2 `category`

- 字段：`id`、`name(varchar64 UNIQUE)`、`status(tinyint)`、`sort_no(int)`、`created_at`、`updated_at`。
- 索引：
  - `uk_name(name)`：防重复分类名。
  - `idx_status_sort(status,sort_no)`：分类列表展示。

#### 3.2.3 `skill`

- 字段：`id`、`name(varchar128)`、`description(text)`、`category_id`、`author_id`、`resource_url(varchar255 not null)`、`status(enum:draft,pending,published,offline)`、`latest_version_id`、`install_count(bigint)`、`avg_rating(decimal(3,2))`、`rating_count(int)`、`published_at`、`created_at`、`updated_at`、`deleted(bit)`。
- 约束：
  - `fk_skill_author(author_id -> user.id)`
  - `fk_skill_category(category_id -> category.id)`
- 索引：
  - `idx_market_status_category(status,category_id,updated_at desc)`：市场筛选 + 最新排序。
  - `idx_market_status_install(status,install_count desc)`：热门排序。
  - `idx_market_status_rating(status,avg_rating desc)`：评分排序。
  - `idx_author_status(author_id,status)`：我的发布列表。
  - `idx_name_prefix(name)`：前缀检索；模糊搜索可补充倒排方案。

#### 3.2.4 `skill_version`

- 字段：`id`、`skill_id`、`version(varchar32)`、`changelog(text)`、`artifact_url(varchar255)`、`created_by`、`created_at`。
- 约束：
  - `uk_skill_version(skill_id,version)`：版本号唯一。
  - `fk_version_skill(skill_id -> skill.id)`。
- 索引：
  - `idx_skill_created(skill_id,created_at desc)`：详情页版本历史。

#### 3.2.5 `install_record`

- 字段：`id`、`user_id`、`skill_id`、`version_id`、`installed_at`、`status(tinyint)`、`source(varchar32)`。
- 约束：
  - `uk_user_skill(user_id,skill_id)`：一人一个 Skill 一条安装记录（更新版本覆盖）。
- 索引：
  - `idx_user_installed(user_id,installed_at desc)`：我的安装列表。
  - `idx_skill_installed(skill_id,installed_at desc)`：统计安装趋势。
  - `idx_version(version_id)`：版本安装分布。

#### 3.2.6 `rating`

- 字段：`id`、`user_id`、`skill_id`、`score(tinyint 1-5)`、`comment(varchar1000)`、`created_at`、`updated_at`。
- 约束：
  - `uk_user_skill(user_id,skill_id)`：一人一评，可更新。
  - `check_score(score between 1 and 5)`。
- 索引：
  - `idx_skill_updated(skill_id,updated_at desc)`：评论列表倒序。
  - `idx_skill_score(skill_id,score)`：评分分布统计。

#### 3.2.7 `audit_record`

- 字段：`id`、`skill_id`、`reviewer_id`、`action(enum:submit,approve,reject,offline)`、`reason(varchar500)`、`created_at`。
- 索引：
  - `idx_skill_created(skill_id,created_at desc)`：审核历史。
  - `idx_pending(created_at desc)`：待审核列表。

### 3.3 分库分表策略

- V1：**不分库分表**，优先单库 + 读写分离。
- 触发阈值建议：
  - `install_record` > 5000w、`rating` > 2000w 后评估分表。
- 拆分维度：
  - `install_record` 按 `skill_id hash` 或按月分区；
  - `rating` 按 `skill_id hash` 分表。
- 路由规则：ShardingSphere（后续），应用层封装路由键。

### 3.4 数据一致性方案

- V1 单体内事务：Spring `@Transactional` 本地事务保证强一致。
- 跨模块异步一致性（安装计数、评分均值、运营统计）：
  - 使用 **Outbox + MQ** 最终一致；
  - 幂等消费键：`event_id`。
- 若未来拆微服务：优先 **Saga**（编排式）而非 TCC  
  - 理由：安装/评分链路补偿简单，TCC 开发和维护成本高；Seata AT 仅在严格 SQL 约束且跨库事务多时考虑。

## 4. 接口设计

### 4.1 RESTful API 规范

- 路径：统一前缀 `/v1`，资源名复数，如 `/v1/skills`。
- 方法语义：
  - `GET` 查询、`POST` 创建、`PUT/PATCH` 更新、`DELETE` 删除/下架。
- 响应结构：
  - 成功：`{success:true, code:"OK", message:"", data:{...}}`
  - 失败：`{success:false, code:"AUTH_TOKEN_EXPIRED", message:"...", data:null}`
- 状态码：
  - `200/201/204/400/401/403/404/409/422/429/500` 按规范映射。
- 请求头：
  - 使用 `HttpOnly Cookie` 携带会话凭证（`Authorization` 仅作为调试兜底，不作为主链路）；
  - `X-Request-ID` 必传；
  - 关键写操作传 `Idempotency-Key`；
  - 所有写操作（`POST/PUT/PATCH/DELETE`）必须携带 CSRF Token（上线硬门禁）。

### 4.2 核心接口清单

| 接口路径 | 方法 | 功能描述 | 入参 | 出参 | 复杂度 |
|---------|------|---------|------|------|--------|
| `/v1/auth/register` | POST | 用户注册 | username, password | userId | 中 |

| `/v1/reviews/pending` | GET | 待审核列表 | page,size,status,keyword | Page\<ReviewItem\> | 中 |
| `/v1/reviews/{reviewId}/approve` | POST | 审核通过 | 可选备注 | 生效状态 | 中 |

### 4.3 异常流程覆盖

- 登录失败：`401 AUTH_INVALID_CREDENTIALS`。
- Token 过期：`401 AUTH_TOKEN_EXPIRED`，前端跳转登录并保留 `returnUrl`。
- 非作者编辑：`403 PERMISSION_DENIED`。
- 版本号重复：`409 SKILL_VERSION_CONFLICT`。
- 资源地址缺失：`422 SKILL_RESOURCE_URL_REQUIRED`。
- 未安装评分：`422 RATING_REQUIRES_INSTALL`。
- Skill 已下架且未安装过：`409 SKILL_OFFLINE_NOT_INSTALLABLE`。
- Skill 已下架：`409 SKILL_OFFLINE_NOT_INSTALLABLE`（历史版本同样不可安装）。
- 幂等冲突：`409 IDEMPOTENCY_CONFLICT`。
- CSRF 缺失或无效：`403 AUTH_CSRF_INVALID`。
- 限流触发：`429 RATE_LIMIT_EXCEEDED` + `Retry-After`。

### 4.4 接口安全设计

- 鉴权：采用 `HttpOnly Cookie` 会话方案（服务端签发、轮换与失效），不要求前端持久化 Bearer Token。
- 授权：RBAC（USER/ADMIN）+ 资源归属校验（author_id）。
- CSRF：双提交 Cookie（或等价方案）+ Header 校验，作为发布硬门禁。
- 防重放：关键写操作校验 `Idempotency-Key` + 时间窗口签名（可选）。
- 限流：网关层（IP/User 令牌桶）+ 应用层（接口维度 Sentinel）。
- 输入安全：Bean Validation + MyBatis `#{}` 参数化 + 输出转义防 XSS。

### 4.5 API 版本管理

- 所有接口挂载 `/v1`，新增字段保持向后兼容。
- 不兼容变更发布 `/v2`，旧版至少保留一个迭代周期并标注 `deprecated`。

## 5. 核心流程时序图（文字）

### 5.1 用户获取安装命令

```text
用户 -> 网关: POST /v1/skills/{id}/install-command (Idempotency-Key)
网关 -> 鉴权服务: 校验 Token
鉴权服务 --> 网关: 用户身份与角色
网关 -> install-service: 生成短时签名安装链接
install-service -> skill-service: 校验 Skill 状态(必须已上架)
skill-service --> install-service: Skill 元数据 + latestVersion + 仓库地址
install-service -> MySQL: 记录命令签发与链接过期时间/单次消费状态
install-service --> 网关: 返回脱敏安装命令（signed-git-url）
网关 --> 用户: 复制命令到剪切板
```

### 5.2 用户评分/修改评分

```text
用户 -> 网关: PUT /v1/skills/{id}/ratings
网关 -> rating-service: 请求转发
rating-service -> install-service: 校验是否已安装
install-service --> rating-service: 已安装=true
rating-service -> MySQL: INSERT ... ON DUPLICATE KEY UPDATE rating
rating-service -> MySQL: 更新 skill.avg_rating/rating_count
rating-service -> Redis: 失效详情与评分列表缓存
rating-service --> 网关: 评分成功
网关 --> 用户: 返回最新平均分
```

### 5.3 作者提交审核 -> 管理员通过

```text
作者 -> skill-service: 提交审核
skill-service -> MySQL: skill.status=draft->pending
管理员 -> review-service: 审核通过
review-service -> MySQL: audit_record(action=approve)
review-service -> skill-service: 状态迁移 pending->published
skill-service -> Redis: 失效市场缓存
skill-service --> 管理员: 审核完成
```

## 6. 非功能性需求评估

### 6.1 性能目标（V1 建议）

- 峰值 QPS：读 800，写 150（内部平台中等规模假设）。
- RT 目标：
  - 列表查询 `P95 < 300ms`；
  - 详情查询 `P95 < 400ms`；
  - 安装/评分写入 `P95 < 500ms`。
- 吞吐：
  - MQ 事件处理能力 >= 3000 msg/min。

### 6.2 并发处理与异步

- 线程池建议：
  - IO 密集：`core=2N`，`max=4N`，队列按接口峰值 2~3 秒容量估算。
  - 统计异步任务独立线程池，避免抢占业务线程。
- 异步方案：
  - `@Async + CompletableFuture` 处理缓存刷新、通知、统计回填；
  - 核心写链路不依赖异步成功返回，保证主流程稳定。

### 6.3 缓存策略

- Key 设计：`skillsops:{domain}:{bizKey}`。
- TTL 建议：
  - 市场列表：30~60s；
  - 详情页：30s；
  - 热门榜单：120s；
  - 会话：30min。
- 防护策略：
  - 穿透：布隆过滤器 + 空值短 TTL；
  - 击穿：热点 Key 互斥锁 + 逻辑过期；
  - 雪崩：TTL 随机抖动 + 多级缓存预热。

### 6.4 降级熔断与限流

- Sentinel 规则：
  - 外部依赖错误率 > 50% 触发熔断；
  - 统计服务超时时降级返回“基础指标 + 缓存旧值”。
- 双重限流：
  - 网关：按 IP/User 全局限流；
  - 应用：安装/评分接口单用户限频（如 10 次/分钟）。
  - 认证：登录接口按 IP+账号双维度限频（防爆破）。

### 6.5 数据安全

- 密码：统一 HTTPS 传输 + 后端 BCrypt/Argon2 存储（前端不做自定义哈希）。
- 敏感字段：日志脱敏（Token、密码、Cookie）。
- SQL 注入：严禁 `${}`，统一 `#{}`。
- XSS：评论内容入库前清洗，出参统一转义。
- 审计：关键行为（审核、下架、权限拒绝）记录审计日志。

## 7. 风险与待确认事项

### 7.1 技术风险

1. **实时刷新成本**：安装量/评分“实时”要求可能放大写放大与缓存抖动。  
   - 替代方案：接口写成功后局部实时，榜单/趋势允许秒级延迟（3-10s）。
2. **市场关键词检索精度**：MySQL `LIKE` 在数据增长后性能下降。  
   - 替代方案：分阶段引入 ES 或 MySQL 全文索引。
3. **审核中状态锁定要求**：若未严格执行“审核中不可编辑”，会出现状态错配。  
   - 方案：`pending` 状态拒绝编辑，审核拒绝后再开放编辑。
4. **版本发布默认无需复审风险**：存在内容治理压力。  
   - 方案：保持“默认无需复审”策略，同时补充审计追踪与异常版本快速下架通道。

### 7.2 依赖风险

- MQ、Redis 不可用时，安装/评分后置任务延迟。
- 第三方仓库地址可达性不稳定（影响安装成功率与用户体验）。
- 监控链路缺失时无法及时发现评分/安装失败峰值。

### 7.3 已确认产品决策（评审冻结）

1. 登录凭证采用 `HttpOnly Cookie`。
2. 注册不校验邮箱验证码。
3. 评分允许“只打分不评论”。
4. 安装流程不需要二次确认。
5. 运营统计时间粒度支持日/周/月。
6. 下架 Skill 后已安装用户仅可查看，不可安装任何历史版本。
7. 版本发布默认无需复审。
8. CSRF 防护为上线硬门禁。
9. 分页参数统一为 `page/size`。
10. 安装命令统一采用 `npx skills add <signed-git-url>`（短时签名链接方案）。

## 8. 开发计划建议

### 8.1 里程碑拆分（建议 4 个迭代）

- **迭代 1（P0，1.5 周）**：认证、权限、Skill 基础 CRUD、状态机、市场列表/详情查询。
- **迭代 2（P0，1 周）**：审核流、安装链路、我的安装、幂等与审计日志。
- **迭代 3（P1，1 周）**：评分评论、分类管理、缓存与限流策略落地。
- **迭代 4（P1，1 周）**：运营统计、监控告警、性能压测与优化。

### 8.2 模块依赖关系

1. `auth` 与 `rbac` 先行（所有受限接口依赖）。
2. `skill/version/review` 其次（发布-审核-上架主链路）。
3. `market/install/rating` 并行推进（面向用户高频路径）。
4. `ops` 最后接入（依赖事件与数据沉淀）。

### 8.3 联调计划

- 节点 A：认证 + 市场检索接口联调（前后端契约冻结）。
- 节点 B：详情安装与异常流程（下架、未登录、幂等冲突）联调。
- 节点 C：评分更新与均分实时刷新联调。
- 节点 D：作者发布、管理员审核、运营统计全链路联调。
- 上线前：压测（读写峰值 + 缓存失效风暴 + 限流场景）与故障演练（Redis/MQ 降级）。
