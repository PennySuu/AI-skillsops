# SkillsOps 全链路架构师评审报告

## 1. 评审概览

### 1.1 评审范围

- 业务原始需求：`doc/需求文档.md`
- 产品交互方案：`doc/产品与交互文档.md`
- 后端技术方案：`doc/后端技术评审文档.md`
- 前端技术方案：`doc/前端技术评审文档.md`
- 覆盖链路：认证、发布、审核、市场检索、详情安装、评分评价、运营统计、权限与安全、稳定性与扩展性。

### 1.2 评审方法

- 采用“业务 -> 交互 -> 接口契约 -> 数据模型 -> 非功能性 -> 架构风险”六层检查法。
- 核心检查清单：
  - 业务闭环与边界场景完整性
  - 前后端接口字段/类型/状态机一致性
  - 鉴权、授权、CSRF、XSS、供应链风险控制
  - 并发一致性、幂等、缓存一致性与降级方案
  - SLO、容灾、监控告警、压测可执行性
  - 单点故障与演进可行性

### 1.3 问题统计

| 严重程度 | 数量 | 说明 |
|---------|------|------|
| 🔴 阻塞性（Blocker） | 2 | 必须解决，否则项目无法推进 |
| 🟠 高风险（Critical） | 5 | 严重影响质量或进度，需优先处理 |
| 🟡 中风险（Major） | 6 | 影响体验或维护性，建议修复 |
| 🟢 低风险（Minor） | 3 | 优化建议，可后续迭代处理 |

## 2. 业务需求层评审

### 2.1 需求完整性

- 主链路“发布 -> 审核 -> 上架 -> 安装 -> 评分 -> 版本更新”整体闭环成立。
- 边界场景覆盖不完整：缺失安装执行失败后的“回滚/补偿”与“不可达仓库”标准处理。
- 多角色定义存在抽象层冲突：业务上“普通用户可兼作者”，技术实现上出现 `USER/AUTHOR/ADMIN` 三角色并行，边界未冻结。

### 2.2 需求一致性

- 术语不一致：排序维度在文档中同时出现“热门/最新/评分”与“安装量/评分/最新发布”。
- 生命周期语义未形成统一字典：业务中文状态与技术英文枚举未提供强制映射规范。
- 数值规则不完整：平均评分“实时展示”有要求，但未定义舍入位数、展示精度、并发更新口径。

### 2.3 需求可实现性

- 可实现但存在高风险项：若安装行为涉及执行外部脚本，当前文档缺失安全沙箱与供应链校验，存在上线阻断风险。
- 成本收益偏离风险：V1 尚未冻结是否支持“下架后历史版本重装”，技术方案已扩展该能力，可能导致合规与运维复杂度提前放大。
- 第三方依赖（repo/doc 链接可达性、稳定性、可信性）缺乏验收标准。

## 3. 产品交互层评审

### 3.1 交互逻辑评审

- 页面流程闭环基本完整，异常态覆盖较好（401/403/加载失败/空态）。
- “待确认项”在产品文档仍未全量闭环，但后端已“冻结决策”，存在需求管理流程断层。
- 安装成功后的反馈及时性与后端异步聚合策略存在潜在冲突。

### 3.2 数据一致性评审

- 前端采用 Vue Query 失效策略合理，但依赖后端返回口径稳定。
- 并发更新（审核中编辑、评分重复提交）在交互层有描述，后端并发控制尚未形成可执行约束（见问题清单）。
- 缓存一致性策略存在“主流程实时、统计链路最终一致”的混合模型，需明确用户可感知延迟上限。

### 3.3 权限与安全评审

- 页面级权限已定义，按钮级权限有规划，但资源归属（author_id）校验需要前后端统一错误语义。
- 敏感操作（下架、拒绝）有二次确认，符合基本治理要求。
- 前端提出 CSRF Token，后端方案未明确签发与校验机制，存在安全断点。

## 4. 后端技术层评审

### 4.1 架构合理性

- “模块化单体”符合 V1 交付效率目标。
- 服务边界总体清晰，但审核/技能状态迁移分属多模块，需防止隐式循环依赖。
- 数据模型覆盖主业务实体，但部分并发与契约字段定义存在缺口。

### 4.2 性能与扩展性

- 索引与缓存策略较完整，具备初期可用性。
- “实时刷新”与异步聚合并存，需引入延迟容忍 SLA 与回查机制。
- MQ 场景合理，但消息丢失/重复消费虽然提到幂等键，仍需明确死信与重放作业流程。

### 4.3 安全与可靠性

- 鉴权与 RBAC 框架已定义，但 CSRF 与供应链安全未落地。
- SQL 注入与 XSS 有防护原则，但未见统一安全测试门禁。
- 降级熔断策略存在原则，未定义故障演练频率与恢复判定标准。

## 5. 前端技术层评审

### 5.1 技术选型评审

- Vue3 + TS + Vite + Pinia + Query 方案与团队能力匹配，技术路径稳健。
- 组件库、构建、请求层方案可支持快速迭代。
- 对后端契约依赖度高，若字段口径未冻结，返工成本会集中爆发。

### 5.2 性能与体验

- 已提出懒加载、分包、防抖节流、弱网反馈，方向正确。
- 尚缺明确性能验收线（例如首屏时间、关键交互响应阈值）。
- 离线/弱网策略偏轻量，若 VPN 抖动明显，需补充重试退避与幂等兜底规范。

### 5.3 可维护性

- 分层结构与命名规范较清晰，可维护性基础较好。
- “后端 DTO -> 前端 VM”映射机制合理，但需契约测试配套。
- 业务状态枚举和权限策略需要统一 schema 管理，避免多处硬编码。

## 6. 前后端一致性评审（重点）

### 6.1 接口契约一致性

- 发现字段命名不一致：创建 Skill 入参在后端清单使用 `desc`，业务/产品/前端均使用 `description`。
- 分页参数存在口径风险：后端清单 `page,size`，需求与交互未冻结 `pageSize`/`size` 命名。
- 角色模型存在不一致：业务是“用户兼作者”，技术实现是“独立 AUTHOR 角色”。

### 6.2 错误处理一致性

- 错误码总体可对齐，前端已建立统一映射机制。
- 但 CSRF 场景错误码未定义（例如 `AUTH_CSRF_INVALID`），会导致前端无法精准提示。
- 限流和降级提示文案规范未冻结，易导致前后端用户提示不一致。

### 6.3 状态流转一致性

- 生命周期主状态一致，但技术层未显式维护统一状态字典（含可触发动作矩阵）。
- “审核中可编辑且需重新提交”在产品定义明确，后端落地仅停留在风险建议，缺少强制机制。
- 下架后可见性策略基本一致，但“可否重装历史版本”超出原始需求边界。

### 6.4 分页与查询一致性

- 查询参数命名和排序字段值域尚未形成契约文档（OpenAPI/JSON Schema）级冻结。
- 大数据量查询策略在前端提到虚拟滚动，后端未给出游标或稳定分页保障策略。
- 统计接口口径（自然日/滚动窗口）未最终冻结，前后端均标为待确认。

## 7. 非功能性需求评审

### 7.1 性能需求

- 后端有 QPS/RT 目标，前端缺少对应体验指标（LCP、首屏可交互、交互响应）明确阈值。
- 缺失压测通过标准（例如成功率、错误率、缓存命中率下限）。
- 尚未定义“实时刷新”可接受延迟窗口（建议 1-3 秒内）。

### 7.2 可用性需求

- 缺失系统可用性目标（如 99.9% 月度可用性）。
- 缺失容灾指标（RTO/RPO）和跨可用区/备份演练计划。
- 监控项较多，但告警分级、Oncall 和升级路径未定义。

### 7.3 安全需求

- 缺失威胁建模产出（资产、攻击面、控制点、残余风险）。
- 认证方案采用 Cookie 时，CSRF 与会话固定攻击防护未形成闭环。
- 缺失 Skill 供应链安全（来源校验、签名、恶意脚本检测）策略，风险高。

### 7.4 可扩展性需求

- 模块化单体支持短期扩展，但缺少“何时拆分”的量化阈值（仅部分表规模阈值，不含团队与发布维度）。
- 业务规则配置化不足（例如排序策略、风控阈值、审核策略未配置中心化）。
- 多租户与多环境隔离未纳入当前设计，后续跨团队推广会受限。

## 8. 问题清单（逐条）

### 问题编号：[P001]
- **问题标题**：安装命令分发包含明文 Token，存在凭据泄露风险
- **严重程度**：🔴 Blocker
- **涉及文档**：业务需求 / 产品交互 / 后端评审 / 前端评审
- **问题描述**：
  已澄清安装模式为“点击安装 -> 复制 `npx skills add ...` 命令到剪切板，由用户自行在终端执行”，平台不直接执行本地脚本。当前命令示例将访问凭据（Hub Token）直接拼接在 URL 中，存在明文暴露风险（页面源码、日志、剪切板历史、截图、IM 转发、终端历史）。
- **根因分析**：
  安装链路从“执行安全”转为“命令分发安全”后，凭据传递模型未重设计，仍采用长期 Token 内嵌命令方式。
- **影响评估**：
  一旦 Token 泄露，攻击者可未授权拉取内部 Skill 仓库，造成资产外泄与横向风险；且泄露面广、追溯困难，属于上线阻断项。
- **修复建议**：
  1) 禁止在前端和复制命令中暴露长期密钥，改为短时签名安装链接（如 60-300 秒 TTL，单次可用）；2) 安装命令统一为 `npx skills add <signed-git-url>`；3) 服务端校验签名链接后再换取真实仓库访问权限；4) 日志、埋点、错误上报对命令参数全量脱敏；5) 凭据轮换与吊销机制产品化（泄露后可立即失效）。
- **责任人建议**：架构 / 后端 / 前端 / 安全
- **优先级**：立即

### 问题编号：[P002]
- **问题标题**：Cookie 会话方案的 CSRF 契约未闭环（缺失项）
- **严重程度**：🔴 Blocker
- **涉及文档**：后端评审 / 前端评审
- **问题描述**：
  前端文档明确“SameSite Cookie + CSRF Token”，后端文档仅说明 HttpOnly Cookie 与鉴权流程，未定义 CSRF Token 签发、轮换、校验接口与错误码。当前已澄清：CSRF 防护为上线硬门禁。
- **根因分析**：
  安全职责边界未在前后端接口契约中冻结，导致防护策略停留在原则层。
- **影响评估**：
  关键写接口（安装、评分、审核、下架）存在跨站请求伪造风险，无法通过安全评审。
- **修复建议**：
  1) 后端增加 CSRF Token 获取与校验机制；2) 所有变更接口统一校验并返回标准错误码（建议 `AUTH_CSRF_INVALID`）；3) 前端请求层统一注入 CSRF 头并处理失效重试；4) 上线验收门槛：所有写接口（POST/PUT/PATCH/DELETE）CSRF 校验覆盖率 100%，并通过负向测试（缺失/伪造/过期 Token）。
- **责任人建议**：后端 / 前端 / 安全
- **优先级**：立即

### 问题编号：[P003]
- **问题标题**：“实时刷新”目标与异步聚合策略冲突
- **严重程度**：🟠 Critical
- **涉及文档**：产品交互 / 后端评审 / 前端评审
- **问题描述**：
  产品要求安装量与均分“接口成功后页面即时更新”，后端采用 Outbox + MQ 异步聚合和缓存失效，未定义前端显示值与最终统计值的一致性窗口。
- **根因分析**：
  实时性需求未拆分“用户局部实时”与“全局统计最终一致”两类指标。
- **影响评估**：
  用户看到数值回跳或短时不一致，降低信任并引发缺陷争议。
- **修复建议**：
  1) 定义双层口径：详情页局部实时、榜单/趋势最终一致；2) 已决议可接受延迟窗口为 `<= 5s`；3) 前端增加“数据刷新中”提示并在窗口内定时回查。
- **责任人建议**：产品 / 后端 / 前端
- **优先级**：立即

### 问题编号：[P004]
- **问题标题**：角色模型定义冲突（USER 兼 AUTHOR vs 独立 AUTHOR）
- **严重程度**：🟠 Critical
- **涉及文档**：业务需求 / 后端评审 / 前端评审
- **问题描述**：
  业务定义“普通用户可发布 Skill”，而后端采用 `USER/AUTHOR/ADMIN` 三角色模型，前端路由对“作者/管理员”单独鉴权，缺少“作者资格如何授予”的规则。
- **根因分析**：
  业务角色与系统权限模型未在一份权限矩阵中统一。
- **影响评估**：
  易出现“能创建不能编辑/能见菜单不能访问接口”等权限异常。
- **修复建议**：
  1) 已决议移除 `AUTHOR` 全局角色，统一为 `USER/ADMIN + 资源归属(author_id)`；2) 输出页面级/按钮级/API 级权限矩阵；3) 增加契约测试覆盖 403 场景。
- **责任人建议**：产品 / 架构 / 后端 / 前端
- **优先级**：立即

### 问题编号：[P005]
- **问题标题**：接口字段命名不一致（`desc` vs `description`）
- **严重程度**：🟠 Critical
- **涉及文档**：业务需求 / 后端评审 / 前端评审
- **问题描述**：
  创建 Skill 入参在后端接口清单定义为 `desc`，需求/产品/前端统一使用 `description`。同类风险还包括查询参数 `category` 与 `categoryId` 的表达不一致。
- **根因分析**：
  缺少统一 API schema 作为单一事实源（SSOT）。
- **影响评估**：
  前后端联调将出现 400/422 错误、字段丢失与重复适配，影响迭代节奏。
- **修复建议**：
  1) 已决议创建 Skill 统一字段名 `description`，废弃 `desc`；2) 分页参数统一为 `page/size`；3) 以 OpenAPI 冻结命名与类型并在 CI 增加契约校验（前端类型自动生成）。
- **责任人建议**：后端 / 前端 / 架构
- **优先级**：立即

### 问题编号：[P006]
- **问题标题**：密码传输方案设计存在重放风险
- **严重程度**：🟠 Critical
- **涉及文档**：后端评审 / 前端评审
- **问题描述**：
  后端提出“前端加盐哈希传输 + 后端 BCrypt 存储”，但前端未实现对应机制且在 HTTPS 场景下此方案可能形成固定口令摘要重放风险。
- **根因分析**：
  将“传输层加密”与“口令存储哈希”混用，缺乏统一安全基线。
- **影响评估**：
  引入伪安全复杂度，增加实现分歧，且可能降低真实安全性。
- **修复建议**：
  1) 已决议统一采用 HTTPS + 后端 BCrypt/Argon2；2) 前端不做自定义口令哈希；3) 增补登录限频（按 IP + 账号双维度）；4) 补充设备风控与密码策略。
- **责任人建议**：后端 / 安全
- **优先级**：迭代一

### 问题编号：[P007]
- **问题标题**：审核中可编辑场景缺少并发控制落地
- **严重程度**：🟠 Critical
- **涉及文档**：产品交互 / 后端评审 / 前端评审
- **问题描述**：
  现已决议将“审核中可编辑”改为“审核中不可编辑”，需同步调整产品交互、前后端权限与状态机，避免审核过程中的并发冲突与状态错配。
- **根因分析**：
  早期需求将可用性优先于一致性，导致审核流程缺少冻结窗口设计。
- **影响评估**：
  若不统一为“审核中不可编辑”，仍会出现审核结果与内容版本错配，造成错误上架或反复驳回。
- **修复建议**：
  1) 已决议审核中不可编辑，后端对编辑接口在 `pending` 状态直接拒绝（403/409）；2) 前端隐藏编辑入口并在详情/工作台展示“审核中已锁定”；3) 审核拒绝后恢复可编辑并要求重新提交。
- **责任人建议**：后端 / 前端 / 产品
- **优先级**：迭代一

### 问题编号：[P008]
- **问题标题**：下架后“可重装历史版本”与原始需求范围不一致
- **严重程度**：🟡 Major
- **涉及文档**：业务需求 / 产品交互 / 后端评审 / 前端评审
- **问题描述**：
  原始需求仅定义“下架后市场不可见，已安装用户可查看”，此前技术方案扩展为“可重装历史版本”。现已决议：下架后不可安装历史版本。
- **根因分析**：

## 10. 架构优化建议

### 10.1 短期优化（迭代内可完成）

- 冻结 API 契约：字段命名、分页参数、错误码、状态枚举一次性对齐。
- 补齐安全基线：CSRF、会话策略、写接口幂等一致化、安全测试清单。
- 补全缺失接口：待审列表、分类管理全 CRUD。
- 收敛审核状态机：审核中禁止编辑，拒绝后解锁并重新提交。

### 10.2 中期演进（2-3 个迭代）

- 建立“安装执行安全网关”：来源可信验证、包签名、执行隔离、失败回滚。
- 建立契约治理流水线：OpenAPI -> 前端类型生成 -> 合约测试 -> 发布门禁。
- 完善可观测性：定义核心 SLI/SLO、告警阈值、链路追踪标准与周报机制。
- 引入数据口径中心：运营指标、评分口径、时间窗口统一配置化。

### 10.3 长期规划（架构升级方向）

- 按热点拆分服务：`market-read` 与 `install/rating-write` 解耦，降低互相干扰。
- 建立事件驱动分析平台：统一行为事件总线与离线/实时指标计算层。
- 支持组织级扩展：多租户隔离、配置中心、策略引擎（审核/风控可配置）。
- 安全治理平台化：供应链扫描、策略审计、风险看板与自动处置闭环。
