# SkillsOps 前端技术评审文档

## 0. 文档目标与边界

- 目标：将 `doc/需求文档.md`、`doc/产品与交互文档.md`、`doc/后端技术评审文档.md` 与 `openspec/config.yaml` 映射为可落地的前端实施方案，用于研发评审、排期和联调。
- 范围：V1 覆盖认证、市场浏览、Skill 详情与安装、评分评价、作者发布与版本、管理员审核与基础运营统计。
- 基线约束：
  - 技术栈：Vue 3 + TypeScript + Vite + Pinia + ESLint（与规范保持一致）。
  - API 规范：统一响应结构 `{ success, code, message, data }`，版本前缀 `/v1`。
  - 安全约束：会话采用 HttpOnly Cookie，前端不在 LocalStorage 持久化登录凭证。
  - 实时口径：详情页安装量/评分写入后 `<=5s` 内刷新；榜单与趋势允许秒级最终一致。

## 1. 需求理解与技术映射

### 1.1 业务场景概述（前端视角）

平台面向团队内部 Skill 资产运营，前端核心任务是“让用户低成本找到并正确使用 Skill”，并在同一套界面中支持作者发布与管理员治理。主链路如下：

1. 用户登录后进入市场页，按分类/关键词/排序检索 Skill。
2. 用户进入详情页复制安装命令，前端不直接执行本地安装；命令由用户在终端执行。
3. 已安装用户可提交或修改评分，前端实时回显均分与评论列表。
4. 作者在工作台创建 Skill、提交审核、发布新版本并查看审核反馈。
5. 管理员在工作台完成待审核处理、分类管理和运营数据查看。

### 1.2 功能模块拆分（产品功能 -> 前端模块）

- 认证模块：登录、注册、会话失效回跳、权限化路由入口。
- 市场模块：筛选栏、卡片列表、分页、列表错误恢复。
- 详情模块：基础信息、安装区、版本历史、评价区、管理员操作区。
- 工作台模块：
  - 作者视角：我的发布、创建/编辑、提交审核、发布版本。
  - 用户视角：我的安装、版本更新提示、下架标记。
  - 管理员视角：待审核、分类管理、运营统计。
- 通用能力模块：请求层、错误码映射、埋点、权限指令、统一空态/错误态组件。

### 1.3 核心用例识别（重点交互）

- 用例 A：市场三步闭环（筛选 -> 详情 -> 安装）需低延迟、强反馈。
- 用例 B：审核拒绝回退（拒绝理由展示 -> 编辑修订 -> 再次提交）需状态一致。
- 用例 C：一人一评可修改，需防重复提交与乐观更新回滚。
- 用例 D：Skill 下架后“市场不可见但我的安装可见”，页面状态与按钮权限需分支渲染。
- 用例 E：Token 过期时跨页面回登录并保留 `returnUrl`，确保任务不中断。
  - 补充约束：下架后仅可查看，不可安装任何历史版本。

### 1.4 用户设备分析（假设 + 建议）

- 目标用户：团队内部研发/管理员，Desktop 场景为主。
- 建议设备分布（上线前需埋点校验）：
  - PC/Laptop：85%（Windows 为主，macOS 为辅）
  - 平板：10%
  - 手机：5%（主要查看，不建议承载复杂管理操作）
- 浏览器建议支持：
  - Chrome（主支持）、Edge（次支持）、Safari 最新两个大版本。
  - 不支持 IE，低于 Chromium 110 仅“有限可用”。
- 网络环境：
  - 办公网 + VPN 混合，需考虑高延迟与偶发抖动，默认启用重试与弱网反馈。

## 2. 技术栈选型

### 2.1 框架选型

- **Vue 3.4+**（Composition API）
- 理由：
  - 与 `openspec/config.yaml` 的项目规范一致，减少团队认知切换成本。
  - Composition API 在复杂页面（详情、工作台）中更利于逻辑复用和拆分。
  - 生态成熟（Pinia/Vue Router/Vite）且招聘与维护成本可控。

### 2.2 语言选型

- **TypeScript 5.x + strict: true**
- 理由：
  - 后端已有 DTO 契约，TS 可直接承接类型映射，降低联调错误。
  - 对状态机页面（Skill 生命周期）和权限分支有更强的静态约束。
  - 符合规范中“禁止 any、统一类型定义”的要求。

### 2.3 构建工具

- **Vite 5.x**
- 理由：
  - 冷启动和 HMR 性能优于传统 Webpack，适合多页面并行开发。
  - Rollup 产物体系成熟，易于 chunk 策略和静态资源优化。

### 2.4 UI 组件库

- **Naive UI 2.x + 少量业务自研组件**
- 理由：
  - 组件视觉风格简洁统一，默认观感更现代，适合内部平台快速构建“优雅”界面。
  - 对表格、表单、抽屉、分页、消息反馈等后台场景覆盖完整，TS 支持友好。
  - 主题定制能力强，可直接承接产品文档中的色彩与间距规范。
  - 自研用于统一空态/错误态、评分组件封装、SkillCard 等业务高复用模块。

### 2.5 状态管理

- **Pinia**
- 理由：
  - Vue 3 官方推荐，API 简洁，天然 TS 友好。
  - 对角色态、权限态、全局配置等跨页面共享状态管理清晰。

### 2.6 路由方案

- **Vue Router 4 + history 模式**
- 理由：
  - 与现代部署/CDN 刷新回源方案兼容，可获得更友好的 URL。
  - 支持嵌套路由与路由元信息，便于权限守卫实现。

### 2.7 数据请求

- **Axios + TanStack Query(Vue Query)**
- 理由：
  - Axios 负责统一拦截和标准化错误处理。
  - Query 负责服务端状态缓存、失效、重试、乐观更新，减少手写样板代码。

### 2.8 样式方案

- **Sass + CSS Modules（页面与业务组件）+ 全局 Design Tokens**
- 理由：
  - 兼顾团队熟悉度与可维护性。
  - 避免大规模原子类心智负担，同时支持组件级样式隔离。

### 2.9 表单方案

- **Naive UI Form + zod（或内置 rules）**
- 理由：
  - 认证、发布、审核拒绝等表单需求固定，Naive UI 表单体系即可覆盖核心场景。
  - `zod` 与 TypeScript 联动更紧密，能减少运行时校验与类型定义重复。
  - 可与后端 422 错误映射，形成前后端一致校验体验。

### 2.10 其他库

- 图表：**ECharts 5**（运营统计趋势和排行展示）
- 日期处理：**dayjs**
- ID 生成：**uuid**（`X-Request-ID`、`Idempotency-Key`）

## 3. 项目架构设计

### 3.1 目录结构（推荐）

```text
frontend/
  src/
    api/                  # 按业务域封装接口
    assets/
    components/
      common/             # 基础组件
      business/           # 业务组件
    composables/          # use* 组合函数
    layouts/              # 主布局、工作台布局
    router/
    stores/
    types/
    utils/
    views/                # 页面级组件（与路由对齐）
    styles/
    plugins/
```

### 3.2 模块拆分策略

- 按业务域拆分：`auth`、`market`、`skill-detail`、`workspace`、`admin`。
- 每个域内保持“三层”：`api`（数据访问）-> `composables/store`（状态与逻辑）-> `view/components`（UI）。
- 横切能力集中：错误处理、权限、埋点、国际化（若后续需要）。

### 3.3 组件分层

- 基础组件：`BaseTable`、`BaseEmpty`、`BaseErrorBlock`、`BaseConfirmDialog`。
- 业务组件：`SkillCard`、`SkillInstallPanel`、`RatingEditor`、`AuditActionPanel`。
- 页面组件：`MarketView`、`SkillDetailView`、`WorkspacePublishedView` 等。

### 3.4 代码规范

- ESLint：`@typescript-eslint` + `eslint-plugin-vue` + import order。
- Prettier：统一格式化规则（双引号/分号按团队规范统一）。
- Stylelint：约束样式命名与层级，避免过深嵌套。
- 提交门禁：`lint-staged + husky`（`*.ts,*.vue,*.scss`）。

### 3.5 命名规范

- 组件：PascalCase（`SkillCard.vue`）
- Composable：`useXxx.ts`（`useSkillInstall.ts`）
- Store：`useXxxStore.ts`
- API 文件：`skill.api.ts`、`market.api.ts`
- 类型文件：`skill.types.ts`、`common.types.ts`

## 4. 路由与导航设计

### 4.1 路由表设计

| 路由 | 页面 | 权限 | 说明 |
|---|---|---|---|
| `/login` | 登录页 | 公共 | 支持 returnUrl |
| `/register` | 注册页 | 公共 | 注册成功跳转市场 |
| `/market` | Skill 市场页 | 登录 | 默认首页 |
| `/skills/:skillId` | Skill 详情页 | 登录 | 动态路由 |
| `/workspace/published` | 我的发布 | 登录 | 仅资源归属作者与管理员可编辑 |
| `/workspace/installed` | 我的安装 | 登录 | 已安装列表 |
| `/workspace/reviews` | 待审核 | 管理员 | 审核通过/拒绝 |
| `/workspace/categories` | 分类管理 | 管理员 | 分类维护 |
| `/workspace/ops` | 运营统计 | 管理员 | 趋势和排行 |
| `/:pathMatch(.*)*` | 404页 | 公共 | 错误兜底 |

### 4.2 路由守卫策略

- 未登录访问受限路由：跳转 `/login?returnUrl=...`。
- 登录状态访问登录页：重定向到 `/market`。
- 管理员路由通过 `meta.roles` 校验，失败进入 403 页面。
- 详情页读取失败（404/下架不可见）跳转状态页并提供返回市场入口。

### 4.3 导航结构

- 顶部导航：市场、工作台、管理员入口（角色控制显示）。
- 工作台侧边栏：我的发布、我的安装、待审核、分类管理、运营统计（按角色裁剪）。
- 面包屑：详情页与工作台子页面显示，提升回溯效率。
- 可选 Tab：工作台内容区内部使用 Tab/Segmented 进行二级切换。

### 4.4 路由与菜单映射

- 菜单定义来源于统一配置（`menu.config.ts`），字段包含 `path/title/icon/roles`。
- 路由注册与菜单渲染共享同一配置，避免“菜单可见但路由不可达”的不一致。

## 5. 状态管理设计

### 5.1 全局状态

- `authStore`：`userProfile`、`role`、`isAuthenticated`、`sessionExpired`。
- `appStore`：全局 loading、主题、错误提示队列。
- `permissionStore`：当前路由权限与按钮级权限映射。

### 5.2 模块状态

- 市场模块：筛选条件、分页、当前列表快照、列表滚动位置。
- 详情模块：Skill 详情、安装状态、评分弹窗状态、管理员操作状态。
- 工作台模块：我的发布筛选、审核弹窗输入、运营时间粒度。

### 5.3 服务端状态策略（Vue Query）

- 缓存键：
  - `['market', filters, page]`
  - `['skill-detail', skillId]`
  - `['my-installs', page]`
- 失效策略：
  - 安装成功 -> 失效 `skill-detail`、`market`、`my-installs`。
  - 评分成功 -> 失效 `skill-detail` 评论与评分摘要查询。
  - 审核成功 -> 失效 `reviews` 与 `market`。
- 乐观更新：
  - 评分提交时先更新本地均分和我的评分，失败后回滚并提示。

### 5.4 状态持久化

- LocalStorage：仅持久化非敏感 UI 状态（筛选偏好、表格列设置）。
- SessionStorage：短期流程信息（`returnUrl`、一次性草稿）。
- IndexedDB：V1 不强依赖；若后续引入离线草稿/大体量缓存再启用。
- 明确禁止：登录凭证、敏感身份数据落地浏览器可读存储。

// RatingEditor props

### 6.3 复杂组件状态流转（示例：`SkillInstallPanel`）

```text
idle -> installing -> success
                \-> error(retryable)

idle -> offlining(confirming) -> success
                             \-> error
```

- `installing` 阶段禁用按钮并显示 loading，防止重复触发。
- 失败状态保留上下文（skillId/version），允许一键重试。
- 下架动作要求二次确认并填写原因（符合交互文档约束）。

### 6.4 通用组件封装计划

- 第一阶段（P0）：`BaseEmpty`、`BaseErrorBlock`、`BaseAsyncButton`、`BaseConfirmDialog`。
- 第二阶段（P1）：`SkillCard`、`RatingEditor`、`VersionTimeline` 抽象为业务复用组件。
- 约束：业务组件只暴露必要 props，不透传底层 UI 库全部参数，避免 API 污染。

## 7. 接口对接方案

### 7.1 API 封装层

- Axios 实例统一配置：`baseURL`、`timeout`、`withCredentials: true`。
- 请求拦截：
  - 注入 `X-Request-ID`。
  - 关键写请求注入 `Idempotency-Key`（安装、审核、评分）。
  - 所有写请求携带 CSRF Token（上线硬门禁）。
- 响应拦截：
  - 解析统一 envelope。
  - `401 AUTH_TOKEN_EXPIRED` 触发登出流程并跳登录。
  - `403 PERMISSION_DENIED` 触发无权限提示。
- 分页契约：
  - 分页参数统一使用 `page/size`，避免多套命名并存。

```ts
// pseudo config
const request = axios.create({
  baseURL: '/v1',
  timeout: 10000,
  withCredentials: true,
})
```

### 7.2 数据转换层

- 后端 DTO -> 前端 ViewModel 映射放在 `api/mappers`：
  - `SkillDetailDTO` -> `SkillDetailVM`（补充 UI 友好字段，如 `isOfflineLabel`）。
  - `OpsDashboardDTO` -> 图表序列数据结构。
- 原则：页面层不直接消费后端原始字段，避免后端字段波动影响 UI 层。

### 7.3 Mock 方案

- 使用 **MSW**：
  - 本地开发默认启用核心链路 mock（登录、市场、详情、安装）。
  - 联调阶段按模块关闭 mock，保留异常场景 mock（429/500/超时）。
- 辅助：保留少量本地 JSON 用于 Storybook/组件联调（可选）。

### 7.4 联调计划（时序）

1. 认证 + 路由守卫联调（登录态稳定后再并行页面开发）。
2. 市场列表/详情联调（覆盖分页、筛选、排序）。
3. 安装命令/评分联调（重点覆盖票据过期、幂等与异常重试）。
4. 作者发布/审核联调（状态机一致性验证）。
5. 运营统计联调（图表数据维度校验）。

### 7.5 WebSocket / SSE

- V1 非必须；当前“实时刷新”通过“写后主动失效 + 前端回查”满足。
- 若后续增加消息中心（版本更新提醒），优先 SSE（实现轻量，服务端成本可控）。

## 8. 页面实现规划

### 页面名称：登录/注册页

- **对应路由**：`/login`、`/register`
- **页面目的**：建立身份会话并恢复用户原任务路径。
- **核心模块**：
  1. 认证表单：账号密码输入 + 校验。
  2. 回跳逻辑：读取 `returnUrl` 并重定向。
- **交互实现**：
  - 点击登录 -> 表单校验 -> 调用登录 API -> 成功后跳转市场/回跳。
  - 登录失败 -> 展示错误码映射文案并保留输入。
- **数据依赖**：`POST /auth/login`、`POST /auth/register`
- **异常处理**：字段校验态、401 错误态、网络失败重试提示。

### 页面名称：Skill 市场页

- **对应路由**：`/market`
- **页面目的**：支持用户高效发现可安装 Skill。
- **核心模块**：
  1. 筛选栏：分类 + 关键词 + 排序（组合条件）。
  2. 卡片列表：SkillCard 组合 + 分页。
- **交互实现**：
  - 切换筛选 -> 更新查询键 -> 拉取列表 -> 回到顶部。
  - 点击详情 -> 跳转详情页。
- **数据依赖**：`GET /market/skills`
- **异常处理**：骨架屏、空结果页、错误页与重试按钮。

### 页面名称：Skill 详情页

- **对应路由**：`/skills/:skillId`
- **页面目的**：完成安装决策、安装执行、评分反馈。
- **核心模块**：
  1. 基础信息 + 版本历史。
  2. 安装面板（含管理员操作）。
  3. 评论与评分编辑。
- **交互实现**：
  - 点击复制安装命令 -> 生成 `Idempotency-Key` -> 调用安装命令接口 -> 将不含长期凭据的短时签名链接命令写入剪切板。
  - 复制成功后展示提示：`安装命令已复制，命令中含有您的 API Token，请勿直接分享给他人`。
  - 提交评分 -> 校验已安装 -> 乐观更新 -> 成功/失败处理。
- **数据依赖**：`GET /market/skills/{id}`、`POST /skills/{id}/install-command`、`PUT /skills/{id}/ratings`
- **异常处理**：加载失败、未安装评分拦截、下架状态禁用安装。

### 页面名称：我的发布（作者）

- **对应路由**：`/workspace/published`
- **页面目的**：完成 Skill 资产发布与版本维护。
- **核心模块**：
  1. 发布列表与状态标签。
  2. 创建/编辑表单（`resourceUrl` 单字段必填，承载仓库地址/文档链接）。
  3. 提交审核与发布版本弹窗。
- **交互实现**：
  - 提交审核 -> 状态更新为审核中并提示。
  - 审核中状态 -> 禁用编辑入口并提示“审核中不可编辑”。
  - 发布版本 -> 校验版本号 -> 成功后刷新版本信息。
- **数据依赖**：`POST /skills`（含 `resourceUrl` 必填）、`PATCH /skills/{id}`、`POST /skills/{id}/submit-review`、`POST /skills/{id}/versions`
- **异常处理**：版本冲突、权限不足、保存失败重试。

### 页面名称：我的安装（用户）

- **对应路由**：`/workspace/installed`
- **页面目的**：查看安装资产并感知更新状态。
- **核心模块**：
  1. 安装列表（安装版本、最新版本、状态标记）。
  2. 快捷入口（查看详情）。
- **交互实现**：
  - 点击详情 -> 跳转并定位版本区域。
  - 可更新状态 -> 显示强调标记。
- **数据依赖**：`GET /users/me/installs`
- **异常处理**：空态引导、下架标记展示、分页失败重试。

### 页面名称：待审核（管理员）

- **对应路由**：`/workspace/reviews`
- **页面目的**：保证上架质量与审核闭环效率。
- **核心模块**：
  1. 待审列表。
  2. 通过/拒绝动作区。
  3. 拒绝理由弹窗。
- **交互实现**：
  - 通过 -> 提交后局部刷新当前行。
  - 拒绝 -> 校验理由长度后提交。
- **数据依赖**：`GET /reviews/pending`、`POST /reviews/{id}/approve`、`POST /reviews/{id}/reject`
- **异常处理**：提交失败保留输入、重复提交防抖。

### 页面名称：运营统计（管理员）

- **对应路由**：`/workspace/ops`
- **页面目的**：展示平台效果并指导运营动作。
- **核心模块**：
  1. 指标卡片。
  2. 安装趋势图。

- **交互实现**：
  - 切换时间粒度 -> 触发全模块重新拉取。
  - 点击排行项 -> 跳转详情。
- **数据依赖**：`GET /admin/ops/dashboard`
- **异常处理**：局部降级（图表失败不阻断指标卡片）。

## 9. 性能优化方案

### 9.1 加载性能

- 路由级懒加载：`views` 全量使用动态导入。
- Code Splitting：
  - `vendor`（vue/pinia/router）
  - `ui`（naive-ui）
  - `charts`（echarts，按需异步加载）
- 预加载策略：市场到详情常用路径使用 `prefetch`。
- 静态资源：启用 gzip/brotli、图片 WebP、CDN 缓存。

### 9.2 运行时性能

- 市场长列表在大数据量场景切换虚拟滚动方案。
- 搜索框防抖（300ms），分页/排序节流（避免重复请求）。
- Vue 层优化：避免模板中复杂表达式，使用 `computed` 缓存。
- 图片懒加载与占位图，减少首屏阻塞。

### 9.3 缓存策略

- HTTP 缓存：静态资源强缓存 + 业务接口协商缓存（可选）。
- Query 缓存：短 TTL + 精确失效替代粗暴全局刷新。
- PWA：V1 不启用；若后续移动访问增长再评估。

## 10. 响应式与多端适配

### 10.1 断点设计

- `>=1280px`：桌面大屏（3 列卡片）
- `1024px-1279px`：桌面中屏（2 列）
- `768px-1023px`：平板（1-2 列，弱化管理操作）
- `<768px`：手机（保留核心浏览与详情）

### 10.2 布局方案

- 页面骨架：Flex + CSS Grid 组合。
- 管理台列表优先表格横向滚动，避免信息截断。
- 关键交互按钮在窄屏下保持可点击区域 >= 40px。

### 10.3 移动端适配

- 字体与间距采用 token 体系，不建议全站 rem 改造。
- 采用媒体查询控制布局变化，`viewport` 使用标准配置。

### 10.4 跨端方案评估

- V1 不引入 React Native/Taro/UniApp/Flutter。
- 理由：当前核心用户为桌面端，跨端投入无法形成短期 ROI。

## 11. 安全与合规

- XSS 防护：评论与富文本内容严格转义，禁止 `v-html` 渲染未净化内容。
- CSRF 防护：SameSite Cookie + CSRF Token（变更请求强制携带，作为上线硬门禁）。
- 敏感数据处理：
  - 登录凭证仅走 HttpOnly Cookie。
  - 前端不记录密码、Token 到日志或埋点。
- 接口安全：
  - 写操作带 `Idempotency-Key`，防重放和重复提交。
  - 错误提示不泄露后端堆栈。
- 可访问性（a11y）：
  - 关键控件支持键盘导航。
  - 弹窗焦点管理与 Esc 关闭。
  - 表单输入项提供可读 label 与错误提示关联。

## 12. 构建与部署

### 12.1 构建配置

- 环境：`dev/test/staging/prod` 多环境变量隔离。
- 配置项：API 地址、Sentry DSN、埋点开关、功能开关。
- 输出策略：按环境注入版本号与构建时间用于排障。

### 12.2 CI/CD 流程

1. PR 阶段：Lint + Type Check + Unit Test。
2. 合并阶段：Build + 产物扫描 + Preview 部署。

- 推荐：静态资源托管到 CDN/OSS，网关统一反向代理 API。

### 12.4 监控与埋点

- 错误监控：Sentry（关联 `X-Request-ID`）。
- 性能监控：Web Vitals（LCP/FID/CLS）+ 首屏指标。
- 业务埋点：`{page}_{action}_{target}`，重点记录搜索、安装、评分、审核。

## 13. 风险与待确认事项

### 13.1 技术风险

- 实时刷新要求高，可能引发频繁请求与缓存抖动。
- 表格/图表场景对低性能设备存在卡顿风险。
- 第三方组件库升级可能带来 breaking changes。

### 13.2 依赖风险

- 后端字段命名变化若未同步类型，会导致前端渲染异常。
- 运营统计接口若口径未冻结，图表配置会频繁返工。
- 审核流接口时序变化会影响前端状态机实现。

### 13.3 交互文档难点/冲突与替代方案

1. **冲突点：安装是否二次确认**  
   - 交互文档曾标记待确认，后端评审已冻结为“无需二次确认”。  
   - 前端方案：默认直接安装；仅在高风险版本（未来策略）启用可配置确认弹窗。
2. **冲突点：评分是否允许仅打分**  
   - 交互文档待确认，后端已冻结“允许仅打分”。  
   - 前端方案：评论字段改为可选，长度校验仅在有输入时触发。
3. **已冻结：下架后不可安装历史版本**  
   - 前端需区分“市场不可见”与“我的安装可查看但不可安装”双态。  
   - 落地方案：详情页与我的安装统一展示禁用态和原因提示。

### 13.4 待确认清单

- 设计稿是否覆盖手机端完整交互（当前仅有桌面优先规范）。
- 运营统计 TopN、趋势口径（自然日还是滚动窗口）是否最终冻结。
- 是否在 V1 引入消息中心（版本更新提醒）并决定 SSE 时机。

## 14. 开发计划建议

### 14.1 里程碑拆分

- 里程碑 1（P0）：脚手架、路由权限、认证闭环、市场列表。
- 里程碑 2（P0）：详情安装、我的安装、核心异常态。
- 里程碑 3（P1）：作者发布、审核流、版本发布。
- 里程碑 4（P1）：评分评论、分类管理、运营统计。
- 里程碑 5（优化）：性能优化、监控埋点、可访问性与稳定性修补。

### 14.2 依赖关系（建议顺序）

1. 认证与权限先行（所有页面依赖）。
2. 市场与详情并行（用户主链路）。
3. 工作台作者与管理员模块（依赖状态机接口稳定）。
4. 运营统计与体验优化收尾。

### 14.3 联调协作节点

- 前后端：接口契约冻结评审（字段、错误码、分页口径）。
- 前端与设计：关键交互走查（审核拒绝、下架状态、异常态）。
- 前端与测试：主链路 E2E 用例（登录 -> 市场 -> 安装 -> 评分）。
- 发布前：联合压测与故障演练（接口超时、401、429、500）。
